## 2. M0: 技術検証（完了）

### 目的
YaneuraOu.wasmを無料静的ホスティング上のブラウザで動作させられるかを検証する。

### 完了した内容
- メインスレッド直接呼び出し方式の採用（Web Worker入れ子構成は動作しないことが判明）
- USIプロトコル通信（`usi`/`isready`/`position`/`go`/`bestmove`）の疎通確認
- 本番候補エンジン（水匠5・hao）でのWASM実行、評価関数差し替えによるAI人格切り替えの確認
- UIブロック計測（実測17〜18ms、理論値16.7ms比で実用上問題なし）

検証コードは `tools/m0-verification/`（軽量版）、`tools/m0-verification-suisho5/`（本番候補版）。
詳細な検証ログは `docs/CHANGELOG.md` を参照。

## 3. M1: 対局UIの最小構成（完了）

### 目的
RPGの進行要素（章・レベル・戦形解禁等）を後回しにし、まず「初期配置からエンジンと1局戦える」
状態を作る。ここで対局UI⇔エンジンの結線パターンを固めることが目的。

### スコープ
- 盤面表示・駒移動（クリック/タップ操作）
- 合法手判定（既存将棋ライブラリの活用を想定）
- 成り／不成りの選択
- 持ち駒表示・打つ操作
- YaneuraOu.wasmとの対局（`tools/m0-verification/index-mainthread.html`の`initEngine()`パターンを移植）
- 詰み・投了等の勝敗判定と結果表示

### スコープ外（M2以降に回す）
- 戦形（囲い）選択・駒落ち等の開始局面カスタマイズ
- アイテム・スキル（ヒント／待った／敵デバフ）
- 難易度・敵強さパラメータのRPG連携
- ティラノスクリプトとの結線（`docs/SYSTEM_DESIGN.md` 2.1節の連携方式自体は未確定のため、
  M1では対局UI単体をスタンドアロンページとして動作させる）

### 成果物の置き場所
`docs/CLAUDE.md`のディレクトリ構成方針に従い `src/board-ui/`（対局UI）・`src/engine/`（エンジン結線）
配下に実装する。

## 4. M2: RPG進行システム

**状態: 着手中**

### 目的
`docs/SYSTEM_DESIGN.md` 3章のデータ構造（戦形・装備構成・レベル解禁テーブル）を
`data/`配下のJSONとして実装し、M1の対局UIから参照できるようにする。

### スコープ
- 戦形マスタデータ（`formations.json`）と対局開始局面（SFEN）への反映
- 敵データ（`enemies.json`）と強さパラメータ（ノード数・最大思考時間・MultiPV等）の対局UIへの反映
- アイテム・スキル（ヒント／待った／敵デバフ）の実装
- 難易度システム（係数テーブル）の実装
- `docs/DATA_SCHEMA.md` のスキーマに厳密に従うこと

### 現在の実装状況
- `data/formations.json`から戦形を読み込み、開始SFENを盤面とエンジンへ反映
- URLクエリ`?formation=<formation_id>`で使用する戦形を指定（未指定時は`standard`）
- `data/enemies.json`から敵を読み込み、URLクエリ`?enemy=<enemy_id>`で指定
- 敵の`node_limit`を強さの主基準として`go nodes`へ反映し、`max_think_time_ms`を安全上限として適用
- `data/difficulty.json`から難易度係数を読み込み、URLクエリ`?difficulty=easy|normal|hard`で指定
- 難易度の`node_limit_mult`を敵の基準ノード数へ適用し、難易度名を対局画面へ表示
- MultiPVから複数の候補手を取得し、敵の`move_rank`と難易度の`move_rank_max_bonus`を
  反映した範囲から指し手を選択（候補不足時と`bestmove`へのフォールバックにも対応）
- 敵の`nnue_file`を`assets/nnue/`配下の評価関数へ解決して初期化前に読み込み、
  NNUE指定時はHalfKP noeval版、未指定時は通常版WASMを動的に選択
- HalfKPローダー・評価関数取得・エンジン初期化の失敗時は通常版の内蔵評価関数へフォールバック

## 8. 優先順位の考え方

- サーバーサイド処理を前提にしない（無料静的ホスティングの制約は全マイルストーンで絶対条件）
- 各マイルストーンは独立して動作確認できる単位に区切る（M1完了時点で対局単体として遊べる状態を目指す）
- ライセンス保留中の素材（振電3等）に依存する実装は行わない。詳細は `docs/SYSTEM_DESIGN.md` 8章を参照
