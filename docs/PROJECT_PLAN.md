## 4. M2: RPG進行システム

**状態: 完了**

### 目的
`docs/SYSTEM_DESIGN.md` 3章のデータ構造（戦形・装備構成・レベル解禁テーブル）を
`data/`配下のJSONとして実装し、M1の対局UIから参照できるようにする。

### スコープ
- 戦形マスタデータ（`formations.json`）と対局開始局面（SFEN）への反映
- 敵データ（`enemies.json`）と強さパラメータ（ノード数・最大思考時間・MultiPV等）の対局UIへの反映
- アイテム・スキル（ヒント／待った／敵デバフ）の実装
- 難易度システム（係数テーブル）の実装
- `docs/DATA_SCHEMA.md` のスキーマに厳密に従うこと

### 現在の実装状況
- `data/formations.json`から戦形を読み込み、開始SFENを盤面とエンジンへ反映
- URLクエリ`?formation=<formation_id>`で使用する戦形を指定（未指定時は`standard`）
- `data/enemies.json`から敵を読み込み、URLクエリ`?enemy=<enemy_id>`で指定
- 敵の`node_limit`を強さの主基準として`go nodes`へ反映し、`max_think_time_ms`を安全上限として適用
- `data/difficulty.json`から難易度係数を読み込み、URLクエリ`?difficulty=easy|normal|hard`で指定
- 難易度の`node_limit_mult`を敵の基準ノード数へ適用し、難易度名を対局画面へ表示
- 「ふつう」では敵の着手ごとに探索ノード数を基準100%・標準偏差45%として、上振れ幅を
  半分に圧縮して10〜145%の範囲で変動
- MultiPVから複数の候補手を取得し、敵の`move_rank`と難易度の`move_rank_max_bonus`を
  反映した範囲から指し手を選択（候補不足時と`bestmove`へのフォールバックにも対応）
- 敵の`nnue_file`を`assets/nnue/`配下の評価関数へ解決して初期化前に読み込み、
  NNUE指定時はHalfKP noeval版、未指定時は通常版WASMを動的に選択
- HalfKPローダー・評価関数取得・エンジン初期化の失敗時は通常版の内蔵評価関数へフォールバック
- `data/items.json`から恒久スキル「読み筋封じ」を読み込み、準備画面またはURLクエリ`?item=`で
  装備すると、難易度補正と合わせて敵の実効探索ノード数を50%に削減
- ヒントはプレイヤー手番の現局面をAIで探索し、最善手と残り回数を表示
- 待ったは直前のプレイヤー手番へ盤面・指し手・千日手履歴を戻し、残り回数を表示
- `data/level_unlocks.json`を戦形・アイテムマスタと照合し、プレイヤーレベルまでの解禁と
  ヒント・待った上限増加を累積
- スタンドアロン用`?level=<1〜255>`（既定1）を準備画面と対局URLへ引き継ぎ、解禁済みの
  戦形・スキルだけを表示
- URLで未解禁の戦形・スキルを直接指定した場合も、エンジン初期化前に拒否

### 実装済みの解禁制御

1. `data/level_unlocks.json`を読み込み、プレイヤーレベルから解禁済みIDを求める。
2. 準備画面の戦形・スキル候補を解禁状態で絞り込む。
3. URLで未解禁IDを直接指定した場合も拒否する。
4. レベル境界、未知ID、登録漏れ、マスタ間参照をテストする。

M3結線後はlocalStorageの`player_level`と解禁配列を正とする。旧`level`起動パラメータは、
保存データがない初回だけ互換入力として取り込む。

### 完了条件

- 戦形、敵、難易度、アイテム、スキル、解禁条件がマスタデータ駆動で対局開始構成へ反映される。
- プレイヤーレベルに応じ、未解禁要素を画面操作とURL直接指定の両方で使用できない。
- ヒント、待った、敵探索量デバフが仕様どおり動作し、残回数・履歴の整合性が保たれる。
- マスタのスキーマ、重複ID、参照先ID、境界値を自動テストで検証する。
- `SPECIFICATION.md`、`SYSTEM_DESIGN.md`、`DATA_SCHEMA.md`、`CHANGELOG.md`が実装と一致する。
