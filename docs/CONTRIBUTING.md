# CONTRIBUTING

このプロジェクトへの貢献方法・開発ルールをまとめます。一人開発の間も、後から見返せるように
このルールに沿って進めます。

## 1. ブランチ運用

```
main        … 常に動作する状態を保つ
develop     … 開発中の統合ブランチ
feature/xxx … 機能単位の作業ブランチ（例: feature/wasm-engine-integration）
fix/xxx     … 不具合修正
```

- `feature/*`, `fix/*` から `develop` へマージ
- `develop` が安定したタイミングで `main` へマージ（マイルストーン区切り等）

## 2. コミットメッセージ規約

Conventional Commits に準拠します。

```
<type>: <概要>

例:
feat: 敵AIの思考時間パラメータを実装
fix: 待った機能で持ち駒が正しく戻らない不具合を修正
docs: システム設計書に復活の呪文の仕様を追記
refactor: 対局UIのエンジン通信部分を整理
chore: 依存ライブラリの更新
```

| type | 用途 |
|---|---|
| feat | 新機能追加 |
| fix | 不具合修正 |
| docs | ドキュメントのみの変更 |
| refactor | 挙動を変えないコード整理 |
| test | テスト追加・修正 |
| chore | ビルド設定・依存関係等 |

## 3. コーディング規約

### JavaScript

- ES Modules を基本とする
- インデントは2スペース
- 将棋関連の変数名は極力SFEN/USI用語に準拠する（`sfen`, `usiMove` 等）、ゲームロジック側の変数名は日本語ローマ字表記より英語表記を優先する

### データファイル（JSON）

- `DATA_SCHEMA.md` に定義したスキーマに従う
- 敵データ・戦形データ・アイテムデータは `data/` 配下にドメインごとにファイルを分割する

## 4. ディレクトリ構成（想定）

```
/data          … 敵・戦形・アイテム等のマスタデータ（JSON）
/src/engine    … 将棋AIエンジン（メインスレッド直接呼び出し）/ USI通信まわり
/src/board-ui  … 対局UI
/src/save      … セーブ・復活の呪文まわり
/src/novel     … ティラノスクリプト連携
/assets        … 画像・音声等の素材
/docs          … 設計書・仕様書（本ドキュメント群）
/tools         … cshogi/python-shogiを用いた開発補助スクリプト（本番配布には含めない）
```

## 5. Issue / PR の粒度

- 1つのIssueにつき、原則1つのマイルストーン内タスク単位で作成する
- PRは`PROJECT_PLAN.md`のマイルストーン番号を参照できるようにタイトルまたは本文に明記する

## 6. ライセンス関連の注意

- 本プロジェクトは YaneuraOu（GPLv3系）を組み込むため、リポジトリ全体を GPLv3 で公開する
- 外部ライブラリ・素材を追加する際は、ライセンスが GPLv3 と両立するか確認し、`ASSETS_CREDITS.md` に追記する
- 支援者限定配布の評価関数ファイル等、再配布不可のファイルはリポジトリに含めない

## 7. ドキュメント更新ルール

- 設計・仕様に変更が生じた場合、実装前に対応するドキュメント（`SYSTEM_DESIGN.md` / `SPECIFICATION.md` / `DATA_SCHEMA.md`）を更新する
- マイルストーン完了時、`CHANGELOG.md` に記録する
