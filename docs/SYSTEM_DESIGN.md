# システム設計書

## 1. 全体アーキテクチャ

```
[Firebase Hosting（無料枠、COOP/COEP応答ヘッダーを設定）]
  │
  ├─ ティラノスクリプト層（演出・進行管理）
  │     - ノベルパート、村の探索、店、サブクエスト、章の分岐
  │     - 名前入力、ゲーム内通貨、定跡書、RPG進行フラグの管理
  │     - 対局UIの呼び出しと結果（勝敗・使用アイテム消費）の受け取り
  │
  ├─ 対局UI層（JavaScript、既存ライブラリ＋自作）
  │     - 外部画像に依存しない自作SVGの盤・駒描画、駒移動・成り選択・持ち駒UI
  │     - 対局の開始局面組み立て（戦形・駒落ち適用後のSFEN生成）
  │     - SFENの駒配置から戦形を検出し、やこび姫のミニキャラと吹き出しで通知
  │     - ヒント／待ったの実行、残数管理
  │     - エンジン（メインスレッド直接呼び出し）との通信（指し手の送受信）
  │
  ├─ 将棋AIエンジン層（メインスレッドから直接実行。詳細は下記5章、`docs/AGENTS.md`参照）
  │     - YaneuraOu.wasm（USIプロトコル）
  │     - 敵データに応じたエンジンオプション設定
  │       （ノード数制限 / 最大思考時間 / MultiPV による次善手選択 等）
  │     - 評価関数（NNUEファイル）の動的ロード
  │
  └─ セーブ／進行管理層
        - localStorage（通常セーブ、自動）
        - 復活の呪文（ブラウザ内の文字列エンコード／デコード、手動発行・入力）
```

ポイントは **サーバーを一切介在させない** こと。すべての処理（将棋のルール判定・AI思考・セーブ）はクライアントで完結する。
名前、経験値、通貨、対局履歴、宝箱、定跡書、サブクエストも外部データベースへ送信しない。
本番配布物は`tools/build-hosting.mjs`で`dist/`へ組み立て、Firebase Hostingから配信する。
リポジトリ全体や開発用ツール、ライセンス未確認の評価関数は配信対象に含めない。

## 2. モジュール間の連携

### 2.1 ティラノスクリプト ⇔ 対局UI

- **iframe埋め込み + 同一オリジン`postMessage`方式を正式採用する。**
  ティラノの`[shogi_match]`タグが全画面iframeを開き、対局UIは独立ページのまま動かす
- 開始構成はiframe URLの`match_id`、`enemy`、`formation`、`difficulty`、任意の`item`で渡す。
  対局UI側では通常の直接起動と同じマスタ・解禁検証を必ず通す
- 終局時は対局UIから親へ、次のバージョン付きメッセージを1回だけ送る。親は
  `event.origin === location.origin`、`event.source === iframe.contentWindow`、`match_id`の一致を検証する

```js
{
  type: "shogi-rpg:match-result",
  version: 1,
  match_id: "chapter1.training:1",
  match: {
    enemy_id: "training_partner",
    formation_id: "standard",
    difficulty_id: "normal",
    item_id: null
  },
  result: {
    outcome: "win", // "win" | "loss" | "draw"
    reason: "checkmate",
    move_count: 57
  }
}
```

- `reason`は`checkmate`、`resignation`、`entering_king`、`repetition`、
  `perpetual_check`のいずれかとする
- iframe内の再読込とリトライではURLと`match_id`を維持する。進行は終局メッセージ受信後にだけ更新し、
  勝利した敵は`shogi_rpg_save.defeated_bosses`へ重複なしで保存する
- ティラノ側は直近結果を`f.match_result`へ、章分岐用の勝利フラグと直近結果を
  `sf.shogi_rpg`へ保存してからシナリオを再開する。敗北・引き分けでは勝利フラグを立てない

### 2.2 対局UI ⇔ 将棋AIエンジン

- **M0技術検証の結果、Web Worker内からYaneuraOu.wasmを呼び出す構成（入れ子Worker）は動作しないことが判明したため、
  メインスレッドから直接YaneuraOu.wasmを呼び出す方式を正式採用する**（詳細・実装パターンは`docs/AGENTS.md`
  「WASM将棋エンジン統合時の必須知識」を参照）。エンジン自体が内部でPThread化されているため、
  UIブロックは実測上問題ないレベルであることを確認済み
- 通信は USI プロトコルのテキストコマンドをそのまま、エンジンモジュールの`postMessage()`/
  `addMessageListener()` APIでやり取りする方式を基本とする（Web WorkerのpostMessageとは別物、
  エンジン側WASMモジュールが提供するAPI名が同じだけなので混同注意）

```js
// 対局UI → エンジン
engine.postMessage("position sfen " + currentSfen);
engine.postMessage("go nodes " + enemyConfig.nodeLimit);
// maxThinkTimeMsを超えた場合は、応答不能を避けるため別途"stop"を送る

// エンジン → 対局UI（addMessageListenerのコールバックで受信）
// "bestmove 7g7f" 等のUSI出力をそのまま受け取る
```

### 2.3 敵の強さ制御パラメータ

敵ごとの設定は静的JSONで管理し、以下のパラメータの組み合わせで強さを表現する。

| パラメータ | 説明 |
|---|---|
| `nnue_file` | 使用する評価関数ファイル（自由配布のもののみ） |
| `max_think_time_ms` | 1手あたりの最大思考時間。極端に遅い端末向けの安全上限 |
| `node_limit` | 探索ノード数の上限。端末性能差を抑えるため、敵の強さの主基準とする |
| `move_rank` | MultiPVで得た候補手のうち何番目までからランダムに選ばせるか（次善手・N番目に有力な手のデバフ） |
| `allowed_openings` | 敵が使用できる戦形（定跡データ）のリスト |
| `handicap` | 駒落ち設定（飛車落ち・角落ち等） |

プレイヤー側のスキル（敵の探索ノード数を減らす等）は、対局開始時にこの設定値へ補正をかける形で実装する。
通常は難易度補正後の`node_limit`まで探索させ、`max_think_time_ms`へ到達した場合だけ`stop`を送る。これにより、
端末が高速なほど敵が強くなる問題を抑えつつ、低速端末で思考が終わらない事態を防ぐ。
探索量デバフの実効ノード数は、`敵のnode_limit × 難易度倍率 × スキル倍率`を四捨五入し、
最低1に補正する。装備スキルは`items.json`の`enemy_debuff_nodes`から選択する。

`nnue_file`が指定された場合、対局UIはファイル名を`assets/nnue/`配下のURLへ解決し、
HalfKP noeval版WASMを動的に読み込み、エンジン初期化前の`preRun`で仮想FSの`/nn.bin`へ配置する。
`null`の場合は通常版WASMの内蔵評価関数を使用する。指定ファイルの未配置・取得失敗・空ファイル、
HalfKPローダー取得失敗、またはHalfKPエンジン初期化失敗時は通常版へフォールバックする。
外部評価関数はWASMビルドとNNUEアーキテクチャが一致するものに限り、実ファイルの追加時に
再配布条件を確認する。

```json
{
  "enemy_id": "chapter1_boss",
  "nnue_file": "rezero_eval.bin",
  "max_think_time_ms": 10000,
  "node_limit": 10000,
  "move_rank": { "min": 1, "max": 1 },
  "allowed_openings": ["standard"],
  "handicap": null
}
```

### 2.4 難易度システムとの関係

難易度（やさしい／ふつう／むずかしい）は、上記の敵パラメータに対する **係数テーブル** として実装する。難易度はいつでも変更可能なため、対局開始のたびに現在の難易度設定を参照してパラメータを決定する。

```js
const DIFFICULTY_MODIFIERS = {
  easy:   { node_limit_mult: 0.5, node_limit_stddev_ratio: 0,    move_rank_max_bonus: 2 },
  normal: { node_limit_mult: 1.0, node_limit_stddev_ratio: 0.45, move_rank_max_bonus: 0 },
  hard:   { node_limit_mult: 1.5, node_limit_stddev_ratio: 0,    move_rank_max_bonus: 0 },
};
```

実効ノード数は`enemy.node_limit * difficulty.node_limit_mult`を四捨五入し、最低1ノードに
補正する。さらに敵が指すたびに、`node_limit_stddev_ratio`を標準偏差の比率とする正規分布で
探索ノード数を補正する。平均から標準偏差2個分に制限した後、正の偏差だけを半分に圧縮する。
現在は「ふつう」のみ、基準100%・標準偏差45%として範囲10〜145%で変動させる。
手の実効ランク範囲は`enemy.move_rank.min`から
`enemy.move_rank.max + difficulty.move_rank_max_bonus`までとする。この最大値をMultiPVへ設定し、
範囲内で取得できた候補手から等確率で1手を選ぶ。候補不足時は取得できた最大ランクまで範囲を縮め、
`info multipv`を取得できなかった場合は`bestmove`へフォールバックする。

### 2.5 ヒントと待った

ヒントはプレイヤーの手番中に現局面をエンジンへ送り、`MultiPV 1`の候補または
`bestmove`を最善手として表示する。ヒント探索中は敵の思考開始や連続使用を無効にする。

待ったは「直前のプレイヤーの指し手＋それに対する敵の応手」の2手を取り消し、
プレイヤーの手番開始局面へ戻す。盤面・指し手履歴・千日手判定履歴を同じ時点まで戻し、
取り消した局面が将来の千日手判定へ混入しないようにする。敵思考中と終局後は使用できない。

## 3. 戦形・アイテム・スキルのデータ構造

### 3.1 戦形（囲い）データ

戦形は「完成局面のSFEN」＋「相性情報」を持つマスタデータとして管理する。

```json
{
  "formation_id": "mino_gakoi",
  "name": "美濃囲い",
  "start_sfen": "....（完成局面のSFEN）....",
  "unlock_level": 3,
  "strong_against": ["yagura"],
  "weak_against": ["anaguma"]
}
```

対局中の戦形通知は、`formation-callouts.mjs`が現在のSFENを盤上の駒配置へ変換し、
棒銀、金矢倉囲い、右四間飛車の成立条件を先手・後手の両方について判定する。
表示名と吹き出し文言は`data/formation_callouts.json`を正とし、同一IDは1局につき一度だけ通知する。
キャラクター画像は`assets/characters/`から読み、Hostingビルドが配布物へコピーする。

### 3.2 プレイヤー装備構成（対局開始前の選択）

```json
{
  "formation": "mino_gakoi",
  "handicap_given_to_enemy": null,
  "hints": { "used": 0, "max": 2 },
  "undo": { "used": 0, "max": 3 },
  "enemy_debuff": "node_limit_half"
}
```

### 3.3 レベルアップ／解禁テーブル

```json
{
  "level": 5,
  "unlocks": {
    "formations": ["anaguma"],
    "items": ["hint_ticket"],
    "skill_upgrades": { "undo_max": "+1" }
  }
}
```

対局UIは`level_unlocks.json`を戦形・アイテムの両マスタと同時に読み込み、未知ID、
登録漏れ、重複解禁、各マスタの`unlock_level`との不一致を検証する。現在レベル以下の
エントリを累積して準備画面の候補と対局中補助の上限を決める。URLによる直接起動にも
同じ解禁判定を適用し、画面の絞り込みを迂回できないようにする。

`player_level`はlocalStorageのセーブデータを正とする。旧`level`クエリは保存データが存在しない
初回起動時だけ互換入力として取り込み、対局URLには引き継がない。

## 4. 復活の呪文（セーブシステム）

### 4.1 現行version 1

現在実装済みの16文字形式は、対局途中の局面を含めず、次の最小限のRPG進行だけを圧縮する。

```js
{
  chapter: 5,                 // 6bit
  playerLevel: 12,            // 8bit
  unlockedFormations: 0b0110101, // ビットフラグ
  defeatedBosses: 0b0001111,     // ビットフラグ
  itemCounts: { hint: 3, undo: 2 }, // 各4bit程度
  difficulty: 1,               // 2bit（やさしい/ふつう/むずかしい）
}
```

### 4.2 現行version 1のエンコード／デコード方針

```
状態オブジェクト → ビットパック → チェックサム付与 → 英数字文字列化（紛らわしい文字除外）
```

- `0/O`、`1/I` など判別しにくい文字は文字セットから除外する
- 大文字小文字は区別しない
- 数bitのチェックサムを付与し、入力ミス・改ざんを検出する
- 実装は72 bitの本体にCRC-8を付けた80 bit固定長で、32文字の独自英数字セットにより
  16文字へ変換する。コード上の呼称は`save_code`に統一する

本編実装後もversion 1の16文字は読込可能とし、追加項目を初期値で補完する。

### 4.3 本編用version 2

名前入力、通常敵の初回勝利経験値、通貨、定跡書、宝箱、講座、サブクエスト、一般アイテムを
復元できるよう、`SR2-`接頭辞を持つ可変長形式を追加する。

```text
version 2状態
  → 安定したマスタ順のビット集合と整数へ圧縮
  → プレイヤー名をUTF-8で付加
  → 保存形式版・コンテンツカタログ版・チェックサムを付加
  → 判別しにくい文字を除いた文字列へ変換
```

- プレイヤー名を含むため固定16文字には収めず、画面にコピー操作を用意する
- 対局途中の局面と一時的な演出状態は含めない
- 生成・検証・復元はブラウザ内で完結させ、サーバーへ送信しない
- 詳細な対象フィールドと互換規則は`DATA_SCHEMA.md` 7.2節・8.2節を正とする

### 4.4 セーブ運用

- **一次セーブ**: localStorage（自動保存）
- **二次バックアップ**: 復活の呪文（プレイヤーが任意のタイミングで発行・記録）
- サーバー保存、アカウント同期、ランキング送信は行わない
- 対局中の一時状態（現在の指し手など）は呪文に含めず、章単位の粗い進行状態のみを対象とする
- `src/save/save-state.mjs`がバージョン移行とマスタ参照検証、`save-storage.mjs`が
  localStorage、`save-code.mjs`がビットパックとCRC-8を担当する

## 5. 将棋AIエンジン導入方針

### 5.1 採用エンジン

- 内蔵評価用に`yaneuraou.wasm` 0.1.2、HalfKP評価関数用に
  `@mizarjp/yaneuraou.halfkp.noeval` 7.6.3-alpha.0を採用する
- 対局開始時に敵の`nnue_file`を見てローダーを選び、初回ロード時にだけ必要なWASMを取得する
- ライセンスはGPLv3系（Stockfish由来コードを含む）。自作コードと結合して配布する際はエンジン部分のソースコード開示義務に留意する

### 5.2 評価関数ファイル

- 自由配布が明言されているもの（リゼロ評価関数など）のみを同梱する
- 支援者限定配布の評価関数（水匠6以降等）は同梱不可のため、敵の強さの階層はエンジンオプション（ノード数制限を主軸とし、最大思考時間・MultiPVによる手のランク制御を補助とする）の組み合わせで表現することを基本方針とする
- **配布物に含める評価関数は「水匠5」「Hao（Háo）」「リゼロ評価関数」の3種**に限定する（詳細は`ASSETS_CREDITS.md`参照）
- **振電3（Shinden3）は暫定的に配布物へ含めない**。開発者本人が無償公開の意図を明言しているものの、正式なライセンス文書（再配布可否等の明文化）が存在しないため、開発者への直接確認が取れるまでローカル検証専用の扱いとする（対応方針は8章を参照）

### 5.3 パフォーマンス上の注意

- **（訂正、2026-07-12）** 当初「Web Worker上での実行を必須」としていたが、M0検証の結果、
  メインスレッドから直接呼び出す方式を正式採用した（2.2節参照）。エンジン内部が既にPThreadで
  マルチスレッド化されているため、この方式でもUI描画のブロックは実測上ごく僅か（理論値16.7msに対し
  実測17〜18ms程度）で、実用上問題ないことを確認済み
- 評価関数ファイルは章が進むごとに都度フェッチし、初回ロード時に全ファイルを読み込ませない
- 本番評価関数（水匠5・hao、各64MB程度）は`isready`応答（内部での解析処理）に約1.3〜1.4秒かかることを確認済み。
  対局開始時のローディングUIで吸収する設計とする
- スマートフォンでの動作速度は別途検証が必要（努力目標）

## 6. 開発用ツールの位置づけ（cshogi / python-shogi）

これらはブラウザ実行には使用せず、**開発時の補助ツール**として位置づける。

- 戦形（完成局面）のSFENデータ作成・検証
- 敵AIの強さパラメータ（ノード数・最大思考時間・手のランク）のチューニング時の対局シミュレーション
- 定跡データ・囲いの相性表を整理するための棋譜解析

## 8. 暫定的にライセンス保留となっている素材の取り扱い

### 8.1 振電3（Shinden3）評価関数

- 現状: 開発者（たややん氏）本人がYouTube動画上で無償公開の意図を明言しているが、正式なライセンス文書（LICENSEファイル・CC表記等）が付与されていない。第三者による再配布・組み込み配布の可否は未確認
- 暫定対応:
  - `shogiAI/Shinden3/` 配下のファイル（実行ファイル・評価関数`nn.bin`）は **リポジトリの本番配布経路（ビルド成果物・Webデプロイ）に含めない**
  - `tools/` 配下、またはgit管理外のローカル検証用ディレクトリに隔離し、**開発者のローカル環境でのみ**動作検証・強さチューニングの参考に使う
  - `.gitignore` で該当パスを除外し、誤ってコミット・公開されることを防ぐ
  - WASMビルド対応コード（`source/eval/nnue/wasm_simd.h/cpp`、`Makefile`のWASMビルド設定）は評価関数本体とは別物であり、**ビルド手法の学習・検証目的でのソースコード参照は問題ない**（ソースコード自体のライセンスはGPLv3系のやねうら王本体に準ずる）。ただし同梱される`nn.bin`（学習済み重み）は上記の理由で配布不可
  - 開発者への利用許諾確認が取れ次第、`ASSETS_CREDITS.md`と本項目を更新し、正式に配布物へ組み込む
- 確認が取れるまでの代替: 敵の強さの階層は「水匠5」「Hao（Háo）」「リゼロ評価関数」の3種＋エンジンオプション（ノード数制限・最大思考時間・MultiPV）の組み合わせで表現する（マイルストーン0〜2の実装はこの前提で進める）

## 9. 七村RPG進行層

`src/rpg/`は名前入力、村移動、会話、対局選択、報酬、経験値、棋貨、店、宝箱、
サブクエスト、定跡集、設定、復活の呪文、エンディングを担当する静的SPAである。
`src/rpg/world-state.mjs`が`data/world.json`と各マスタの参照を検証し、報酬の重複防止と
レベル解禁を一元管理する。対局自体は`src/board-ui/`をiframeで再利用する。

本編と対局UIは共通の`shogi_rpg_save` version 2を使用するため、対局中に消費したヒント・
待ったも村へ戻った時点で反映される。Hostingのルート入口は七村RPGとし、従来のティラノ入口は
`/src/novel/index.html`、単独対局は`/src/board-ui/index.html`から引き続き利用できる。
