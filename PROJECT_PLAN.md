# プロジェクト計画書

## 1. 開発方針

技術リスクが最も高い「ブラウザ内での将棋AI実行」を最優先で検証し、実現可能性を早期に確定させる。
その後、リスクの低い部分（RPG進行・演出）を積み増していく順序で開発する。

理由：個人開発では作りやすい部分（RPG進行等）から着手しがちだが、根幹となる技術検証を先送りにすると
後工程で構成全体の見直しが必要になるリスクが大きいため。

## 2. マイルストーン

### マイルストーン0: 技術検証（最優先）

- [ ] `YaneuraOu.wasm` をローカル環境で動作させる
- [ ] Web Worker上でUSIコマンドの送受信ができることを確認する
- [ ] 自由配布の評価関数（リゼロ評価関数等）で対局が成立することを確認する
- [ ] Chrome（PC）での動作速度を計測する
- [ ] 思考時間・ノード数制限・MultiPVによる強さ調整が機能することを確認する
- [ ] （余力があれば）スマートフォンのChromeでの動作を確認する

**目的**: この段階で致命的な問題（動作しない・速度が実用に耐えない・ファイルサイズが非現実的）が
見つかった場合、AIエンジンの選定や構成自体を見直す。

### マイルストーン1: 対局UIの最小構成

- [ ] 既存の将棋盤JSライブラリの選定・調査
- [ ] 盤面表示・駒移動・成り選択・持ち駒UIの実装（ライブラリで不足する部分を自作）
- [ ] Web Workerとの通信を組み込み、人間 vs AI の対局が最低限成立する状態にする
- [ ] 開始局面を任意のSFENから指定できるようにする（戦形・駒落ち適用のため）

### マイルストーン2: 敵の強さ制御システム

- [ ] 敵データ（JSON）による思考時間・ノード数・手のランク制御の実装
- [ ] 難易度（やさしい／ふつう／むずかしい）による係数テーブルの実装
- [ ] 難易度をゲーム中にいつでも変更できるUIの実装
- [ ] cshogi/python-shogiを用いた開発側での強さチューニング・検証

### マイルストーン3: 戦形・アイテム・スキルシステム

- [ ] 戦形（囲い）マスタデータの作成（完成局面SFEN、解禁レベル）
- [ ] 戦形同士の相性表の設計
- [ ] アイテム／スキル（駒落ち付与、最善手ヒント、待った、敵デバフ、戦形開始、持ち駒増加）の実装
- [ ] レベルアップ／章進行に応じた解禁テーブルの実装

### マイルストーン4: セーブシステム（復活の呪文）

- [ ] localStorageによる自動セーブの実装
- [ ] 状態のビットパック設計の確定
- [ ] エンコード／デコード関数の実装（チェックサム付き）
- [ ] 呪文の発行・入力UIの実装

### マイルストーン5: RPG進行（ティラノスクリプト）

- [ ] ティラノスクリプトでのノベルパート・章分岐の実装
- [ ] 対局UIとの連携（起動・結果受け取り）の実装
- [ ] 難易度変更・準備（アイテム収集/装備）画面の実装
- [ ] 勝敗後のリトライ／拠点帰還フローの実装

### マイルストーン6: 素材・演出

- [ ] フリー素材の選定（駒・盤・キャラ・マップ）
- [ ] 不足素材の自作
- [ ] ライセンス表記（クレジット）の整備

### マイルストーン7: 統合・調整・公開

- [ ] 全体の結合テスト
- [ ] 難易度バランス調整
- [ ] 無料ホスティングへのデプロイ（ファイルサイズ・帯域の確認）
- [ ] GPLv3を含むライセンス表記・ソース公開対応の整備

## 3. 依存関係の概観

```
M0（技術検証） ──┬─→ M1（対局UI） ─→ M2（敵の強さ制御） ─┐
                 │                                        ├─→ M7（統合・公開）
                 └─→ M4（セーブ）    M3（戦形/アイテム）  ┤
                                     M5（RPG進行）        │
                                     M6（素材・演出）  ───┘
```

M0が通らない場合は構成の見直しが必要になるため、他マイルストーンの本格着手前に完了させる。

## 4. リスクと対応

| リスク | 影響度 | 対応方針 |
|---|---|---|
| WASM将棋エンジンの速度が実用に耐えない | 高 | M0で最優先検証。ダメな場合は評価関数を軽量なものに絞る、または探索深さの上限を厳しくする |
| 評価関数ファイルの自由配布版だけでは強さの幅が狭い | 中 | 思考時間・ノード数制限・MultiPVの組み合わせで幅を作る。将来的にライセンス確認の上で追加検討 |
| GPLv3のソース開示義務 | 中 | 開発当初からOSSとして公開する前提にしておけば問題になりにくい |
| スマホでの動作が不十分 | 低〜中 | 「できれば対応」の位置づけとし、PC版の完成を優先する |
| 素材のライセンス・入手性 | 低〜中 | フリー素材を優先探索し、不足分のみ自作に回す（M6を後半に配置） |

## 5. 優先度の考え方

1. 動くかどうかが不確実な技術要素（M0）
2. ゲームの根幹となる対局システム（M1, M2）
3. RPGとしての差別化要素（M3, M4）
4. 演出・体験の質（M5, M6）
5. 仕上げ（M7）
